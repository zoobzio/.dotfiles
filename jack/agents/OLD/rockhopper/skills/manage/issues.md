# Issues

GitHub issue management — commenting on and augmenting existing issues. Rockhopper does not create issues — issues are provided to the team.

## Commenting on Issues

All comments go through the ROCKHOPPER protocol. Run `/protocol` before posting.

### Comment Types

| Type | When | Content |
|------|------|---------|
| Scope | `/scope` posts requirements | Requirements, acceptance criteria, scope boundaries |
| Board summary | `/commission` posts board | Task list, dependencies, support signal |
| Status update | Phase transitions | What phase we're entering and why |
| RFC response | After evaluating an RFC | Decision, rationale, next steps |
| Test summary | Kevin posts after Build | Coverage, findings, pass/fail |
| Architecture plan | Fidgel posts during Plan | Spec, affected areas, approach |

### Commenting Principles

- One comment per purpose — don't combine unrelated updates
- Factual, not narrative — state what happened, not the story of how
- Structured — headers, tables, checklists for scannability
- No agent names — ROCKHOPPER voice only

## Augmenting Existing Issues

When an issue exists but is incomplete:

1. Read what's there — understand the original intent
2. Preserve the motivation — don't rewrite someone else's "why"
3. Add what's missing — requirements, criteria, scope boundaries
4. Sharpen what's vague — replace ambiguity with precision
5. Comment on changes — explain what was added and why
